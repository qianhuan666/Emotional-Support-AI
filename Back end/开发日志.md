我这次更新主要是彻底将前端聊天从单次请求响应（SSE模式）重构为了WebSocket双向长连接模式。

具体变更清单如下：

1. 增加安全的连接机制
后端 API (api/ai-chat.jsp): 新增了 action=wstoken 接口。它会携带系统内部的 APP_MAIBOT_TOKEN 去向 MaiBot 申请一个 60 秒有效的临时 WS token，并带回 ws://.../api/chat/ws 地址及 userId。这避免了直接把高权限的系统密钥暴露在前端浏览器中。
2. 移除旧的 SSE 通信代码
删除了 ai-chat.js 中复杂的 AbortController、TextDecoder 和 fetch(CHAT_URL) 逻辑。
删除了手动拼凑 chunk 的逻辑。
3. 实现 WebSocket 双向长连接
在 ai-chat.js 中加入原生的 new WebSocket() 连接管理代码：
自动连接：页面一加载就会去请求临时 Token 并建立 WebSocket 通道。
状态管理：你在右侧看到的"已连接到 MaiBot / 正在连接 MaiBot..."，也是这个时候加上去的，按钮会自动根据连接状态置灰或启用。
掉线重连：当连接因为网络原因断开时，它会自动在后台每隔几秒重新发起连接（采用指数退避算法，也就是 2秒、4秒、8秒这样往上加），保证不会永久掉线。
4. 接入 MaiBot 的事件分发体系
现在的 ai-chat.js 不再只是发送一条消息等一条回复，而是变成了“监听员”角色。当建立 WebSocket 后，它能处理以下服务器推送来的各种事件：

bot_message: AI 回复的消息。不仅能收到你提问后 AI 的回答，最重要的是现在可以收到 MaiBot 未经你触发而主动发来的对话了。
history: 初始化连接成功后，会自动把历史消息加载出来。
user_message: 当你的其他设备也在连接同一个对话群组并发消息时，这边能同步显示出来（不重复显示自己发的）。
session_info: 用于标识你的身份。
error: 将系统报错以 AI 对话的形式展现。
5. 移除了旧有的离线数据库查询逻辑
由于 WebUI WebSocket 模式是由 MaiBot 统一管理其本地 SQLite 历史的，所以我把你之前基于 JSP 查询 MySQL 的复杂 action=list、action=history 等代码移除或屏蔽了，现在全部以 MaiBot 发送的 history 事件为准，极大精简了代码逻辑。
简单来说，这次更新使得 Web 网页和 MaiBot 之间的通信从“打有线电话（拨号、通话、挂断）”变成了“连对讲机（频道常开、随时插话）”，彻底解决了 MaiBot 主动发消息收不到的问题。
